The **SSA UI Kit** is SSA Group's shared React component library — buttons,
forms, tables, navigation, and more, all driven by a single Emotion theme. This
site documents individual components with live, interactive examples rendered
from the real `@ssa-ui-kit/core` package; [Storybook](https://ui-kit-core.web.app)
remains the playground and visual-regression tool.

## Installation

```bash
pnpm add @ssa-ui-kit/core
# or
npm install @ssa-ui-kit/core
# or
yarn add @ssa-ui-kit/core
```

`@ssa-ui-kit/hooks` is an optional, separate package of shared hooks:

```bash
pnpm add @ssa-ui-kit/hooks
```

### Peer dependencies

The kit targets **React 19.x** and requires Emotion and React Hook Form:

```bash
pnpm add react@19.x react-dom@19.x @emotion/react @emotion/styled react-hook-form
```

> Using React 18 may cause compatibility issues — upgrade to React 19 first.

### Vite users

If you see `useFormContext() returned null` errors from form components
(`DatePicker`, `DateRangePicker`, etc.), it's because the UI kit's webpack
build expects the CommonJS build of `react-hook-form`, while Vite defaults to
ESM — two different module instances means two different React contexts.
Alias it in `vite.config.ts`:

```ts
resolve: {
  dedupe: ['react', 'react-dom', 'react-hook-form'],
  alias: {
    'react-hook-form': path.resolve(
      __dirname,
      './node_modules/react-hook-form/dist/index.cjs.js',
    ),
  },
},
```

Webpack-based projects don't need this. See
[react-hook-form#8281](https://github.com/react-hook-form/react-hook-form/issues/8281)
for background.

## Quick start

Wrap your app in `ThemeProvider` from `@emotion/react` with the kit's
`mainTheme`:

```tsx
import { ThemeProvider } from '@emotion/react';
import { mainTheme, Theme as T } from '@ssa-ui-kit/core';

declare module '@emotion/react' {
  export interface Theme extends T {}
}

function App() {
  return <ThemeProvider theme={mainTheme}>{/* your app */}</ThemeProvider>;
}
```

The `declare module` block augments Emotion's `Theme` type — required for
TypeScript projects, and best kept in one shared `types/emotion.d.ts` file.

## A note on server-side rendering

`@ssa-ui-kit/core` currently ships as a single **UMD bundle** that touches
`window` for its charting dependencies (Plotly, Nivo) at import time, so it
can't be evaluated during server-side rendering. In Next.js or other SSR
frameworks, load it dynamically with `ssr: false` — exactly how the live
examples on this site are rendered.
